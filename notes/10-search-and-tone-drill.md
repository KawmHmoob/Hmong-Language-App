# Search & Tone Drill

Two small additions documented together because they're each ~50 lines.

## What
- **Search** (`/search`) — a single-page search across the alphabet, vocabulary, course, and readings. Live filter, results grouped by section.
- **Tone Drill** — a new quiz (`/quiz/tone-drill`) that asks "what tone does this Hmong word have?" — uniquely valuable for a tonal language and not generated by mapping markers (the existing `alphabet-tones` quiz does that).

## Files
- `src/pages/Search.jsx` — search page
- `src/data/toneDrill.js` — sample tone drill word list
- `src/data/quizzes.js` — added `tone-drill` config + dataset case
- `src/components/Navbar.jsx` — added a `⌕` icon link to `/search`

## Why a single search page (not a navbar input)

A navbar input would be marginally faster to access but:
- The navbar is already crowded (6 nav links + 2 badges + 3 icon links).
- Search results need real estate to display grouped by section — a popover would be cramped.
- A dedicated route is shareable/bookmarkable (`/search?q=…` if you ever add the URL param).

A page costs nothing extra; a navbar input would cost layout.

## Why a static index built at module load

```js
const INDEX = buildIndex()
```

The `INDEX` array is built **once when `Search.jsx` is imported**, not on every search keystroke. Reasons:

- All source data is static — `consonants`, `categories`, `grammar`, etc. are imported at module load too. Building the index once is the natural cache.
- Filtering is cheap (a few hundred items, `String.includes`). No need for a search library.
- If/when data becomes dynamic (e.g. user-added vocab), move the build inside a `useMemo` keyed on the dynamic source.

## Code anatomy: Search

### Index entry shape

```js
{
  kind: 'vocab',                          // groups results in the UI
  label: 'dev',                           // primary text shown
  hint: 'dog',                            // secondary text shown
  to: '/vocabulary/animals/animals-dog',  // link target
  haystack: 'dev dog mammal pet',         // pre-normalized search text
}
```

The pre-normalized `haystack` saves work per keystroke — the filter doesn't have to re-`toLowerCase()` 500 strings on every keystroke.

### Filter + group

```jsx
const grouped = useMemo(() => {
  const norm = normalize(q)
  if (!norm) return null
  const hits = INDEX.filter((i) => i.haystack.includes(norm)).slice(0, 80)
  return hits.reduce((acc, h) => {
    acc[h.kind] = acc[h.kind] || []
    acc[h.kind].push(h)
    return acc
  }, {})
}, [q])
```

Three things worth noting:
- `null` (not `{}`) when empty — the renderer uses `!grouped` to show the prompt vs. results.
- Hard-cap 80 results — long enough to be useful, short enough to never lag.
- `useMemo` keyed on `q` — re-filters only when input changes, not on every keystroke caused by other state.

### Adding a new searchable source

1. In `buildIndex()`, push entries from your new data:
   ```js
   for (const x of myNewData) {
     items.push({
       kind: 'mykind',
       label: x.someField,
       hint: x.otherField,
       to: `/path/to/${x.id}`,
       haystack: `${x.someField} ${x.otherField}`,
     })
   }
   ```
2. Add `mykind: 'My Section'` to the `KIND_LABEL` map.
3. Done — it shows up in results.

## Code anatomy: Tone Drill

### Dataset

`src/data/toneDrill.js` is a flat array:

```js
export const toneDrillWords = [
  { word: 'pob', tone: 'High' },
  { word: 'poj', tone: 'High-falling' },
  // ...
]
```

These are sample words and **need verification by a native speaker**. The note at the top of the file flags this. Add or replace freely — the engine just reads from the array.

### Engine integration

The existing quiz engine is fully generic, so the tone drill is just one new config + one new `case` in the dataset adapter:

```js
// quizzes.js
{
  id: 'tone-drill',
  title: 'Tone Drill',
  description: 'Identify the tone of each Hmong word…',
  questionCount: 12,
  questionTypes: ['multiple-choice'],
  category: 'Tones',
}

// getQuizDataset
case 'tone-drill':
  return toneDrillWords.map((w) => ({ prompt: w.word, answer: w.tone }))
```

That's it. The engine generates 4-option multiple-choice questions (correct tone + 3 distractor tone names) automatically.

### Why this is different from `alphabet-tones`

| Quiz | Prompt | Answer | What it teaches |
|---|---|---|---|
| `alphabet-tones` | `b` (the marker) | `High` | Marker → tone name (memorize the table) |
| `tone-drill` | `pob` (a real word) | `High` | Word → tone name (recognize tones in context) |

The drill is the more valuable practice — recognizing the tone of an actual Hmong word is the real-world skill. The marker quiz is just the prerequisite vocab.

## How to extend

### Make the tone drill audio-based

When real audio exists:
1. Add an `audio` field to each entry in `toneDrillWords`.
2. Update the dataset adapter to thread it through:
   ```js
   return toneDrillWords.map((w) => ({ prompt: w.word, answer: w.tone, audio: w.audio }))
   ```
3. In `QuizEngine.jsx` `MultipleChoice`, read `question.audio` and pass it to `<AudioButton audioSrc={question.audio} />` instead of `null`.
4. Optionally: hide the written word and require the user to identify the tone from audio alone (truer drill).

### Add fuzzy matching to search

For typo tolerance, swap `haystack.includes(norm)` for a fuzzy matcher. A small standalone library (e.g. `fuse.js`, ~10KB) is justifiable here — but the index is already lowercase-normalized, so even a hand-rolled "first-N-chars match" gets you 80% of the value. Try the cheap version first.

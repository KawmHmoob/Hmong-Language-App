# Notebook

## What
A user-owned scratchpad with two sections: **Saved Words** (refs to vocab entries with personal notes attached) and **Notes** (free-form titled notes for reading reflections, vocabulary lists from books, anything). Persisted to localStorage per-user, separate from progress data.

## Files
- `src/context/NotebookContext.jsx` — provider + `useNotebook` hook
- `src/pages/Notebook.jsx` — `/notebook/:tab` page with both sections
- Saved-word entry button lives in `src/components/vocabulary/WordDetail.jsx`

## Why a separate context (not part of ProgressContext)
- **Different ownership model.** Progress is *behavioral telemetry* generated by app actions; notebook entries are *creative content* the user authors. Mixing them muddies what "export my data" or "delete my account" should mean.
- **Different persistence frequency.** Progress changes on every quiz answer (writes are bursty, small). Notes change on save clicks (writes are rare, can be larger). Different contexts let us add a debounce later for one without affecting the other.
- **Easier to add or remove.** The notebook is a feature; if it ever needed to be ripped out (or split into a separate "notes app"), having it isolated is a clean cut.

The downside — one more provider in the tree — was deemed worth it for the separation of concerns.

## Code anatomy

### Shape

```js
{
  savedWords: {
    'animals-dog': { note: "my grandma's dog was named Lus", savedAt: '2026-05-07T...' },
  },
  notes: [
    {
      id: 'note-abc-xyz',         // generated, see newId()
      title: 'From the green book p.42',
      body: 'multi-line text…',
      tags: [],
      createdAt: '2026-05-07T...',
      updatedAt: '2026-05-07T...',
    },
  ],
}
```

### ID generation

```js
function newId(prefix = 'note') {
  return `${prefix}-${Date.now().toString(36)}-${Math.random().toString(36).slice(2, 6)}`
}
```

Time-prefixed so IDs sort naturally; random suffix so two notes created in the same millisecond don't collide. Not cryptographically unique — fine for client-only IDs. Replace with `crypto.randomUUID()` if Supabase ever requires it.

### Mutator pattern

Every mutator follows the same shape: take input → call `setState` with a function that returns the next state. No direct mutation, no async, no side effects beyond `setState`.

```js
const saveWord = useCallback((wordId, note = '') => {
  setState((s) => ({
    ...s,
    savedWords: {
      ...s.savedWords,
      [wordId]: { note, savedAt: new Date().toISOString() },
    },
  }))
}, [])
```

The save-to-localStorage effect at the top of the provider picks up the change automatically:

```jsx
useEffect(() => { save(userId, state) }, [userId, state])
```

So mutators only describe state transitions — persistence is handled once, declaratively.

### Routing pattern

The notebook reuses the same `useParams`-driven tab pattern as Alphabet and Course:

```jsx
// App.jsx
<Route path="/notebook" element={<Navigate to="/notebook/saved" replace />} />
<Route path="/notebook/:tab" element={<Notebook />} />
```

```jsx
// Notebook.jsx
const { tab } = useParams()
{tab === 'saved' && <SavedWords />}
{tab === 'notes' && <Notes />}
```

If you add a third tab (e.g. tags or a calendar view), add it to the `tabs` array at the top and add a render branch — no router changes needed.

### Inline editing pattern

Both SavedWordItem and NoteItem use the same idiom: a local `editing` boolean swaps between display and edit views. Cancel restores original values from props; Save calls the mutator.

```jsx
const [editing, setEditing] = useState(false)
const [body, setBody] = useState(note.body)

const save = () => {
  updateNote(note.id, { title, body })
  setEditing(false)
}

return editing ? <EditView /> : <DisplayView onClick={() => setEditing(true)} />
```

This avoids needing a route for editing — fast to enter and exit, no URL juggling.

## Reading the notebook anywhere

```jsx
import { useNotebook } from '../context/NotebookContext.jsx'

const { savedWords, saveWord, unsaveWord, notes, createNote } = useNotebook()

// Is this word saved?
const isSaved = Boolean(savedWords[word.id])

// Toggle:
isSaved ? unsaveWord(word.id) : saveWord(word.id)
```

The "Save to Notebook" button on `WordDetail` follows exactly this pattern.

## Adding a new field to a note

Say you want notes to have a `color` field for visual organization.

1. In `createNote` in `NotebookContext.jsx`, add a default:
   ```js
   color: data.color || 'cream',
   ```
2. The `{ ...initialState, ...JSON.parse(raw) }` merge in `load()` means existing notes without `color` get `undefined` — handle that in your renderer (`note.color || 'cream'`).
3. Update `NoteItem` to show/edit it.
4. Document it in this note.

## Migration to a backend

Same pattern as `ProgressContext` — replace `load` and `save` with API calls. Saved-word notes and free-form notes could be one table or two; one is simpler:

```sql
create table notebook_entries (
  id uuid primary key,
  user_id uuid references auth.users,
  type text check (type in ('saved_word', 'note')),
  word_id text,                  -- only for saved_word
  title text,
  body text,
  tags text[] default '{}',
  created_at timestamptz default now(),
  updated_at timestamptz default now()
);
```

The two-store-one-shape design in this provider was chosen partly so the eventual API can be either schema.
